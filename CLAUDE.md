# 営業メール自動送信ツール — 開発ルール

> このファイルはリポジトリのルートに `CLAUDE.md` として置く。Claude Code が毎セッション自動で読み込む。

## このプロジェクトは何か

Next.js（App Router）+ SQLite（better-sqlite3）+ Claude API の営業メール作成ツール。
会社URLを貼ると、HPをクロール → AIが企業分析 → その会社向けの営業メールを生成する。

**要件書は `docs/spec-v2.md`。判断に迷ったらこれを正とする。** 調査の根拠は `docs/research/` 配下。

### 収集ジョブの実行主体はアプリ本体（変更しない）

企業リストの常時収集は **アプリ内スケジューラ** が回す（`instrumentation.ts` → `lib/collection-job.ts`）。
外部cronから叩く口も用意してあるが（`POST /api/collection/cron` + `CRON_SECRET`）、**どちらもアプリのDBに書く**。

**PC上のcronがローカルのDBに直接書く構成にはしない。** 本番DBは Railway Volume にあり、
ローカルDBに書いても在庫は本番に反映されない。2026-07-20 に「常時稼働PCでcronを回す」依頼があったが、
同じ理由で「PCのcronがアプリのAPIを叩く」形に変更した。

実行間隔・多重起動の排他は DB に永続化してあるので、再起動しても1日1回を超えない。

### 画面構成は6項目（勝手に増やさない）

ナビは **ダッシュボード / 企業リスト / 生成 / 一括送信 / 履歴 / 設定** の6つ。
作業の順（集める → 書く → 送る → 見る）に並んでおり、右端の「設定」の前だけ区切り線を入れている。

**新しい画面を足すときは、6項目に7つ目を足すのではなく、既存のどれかのタブにする。**
2026-07-20 に10項目まで増えて読めなくなった経緯がある（項目が1つ増えるたびに全部が読みにくくなる）。

| まとめ先 | 中身 | ルート |
|---------|------|--------|
| 企業リスト | 在庫と自動収集 / キーワードで探す | `/collection`, `/collection/search` |
| 設定 | テンプレート / サービス / 人格 / 送信しないリスト / 全般 | `/settings/*`（`/settings` が全般） |

タブは**ページ内の状態ではなくルートで分ける**（`components/tab-nav.tsx`）。
状態で切り替えるとブックマーク・共有・戻るボタンが効かなくなる。

**スマホの下タブは4つ＋「その他」**（`app/nav-header.tsx` の `MOBILE_TAB_HREFS`）。
下タブは5枠までが基準なので、6項目は全部入らない。「生成」をシート側に置いているのは、
URLを貼って長文を直す作業がスマホ向きでないため。

**一括送信のCSV取込は移動しない。** あれは「その送信の宛先を読み込む」機能であって、
企業リストを増やす機能ではない（移すと一括送信で宛先を選べなくなる）。

### 「取得0件」は2種類ある（混同しない）

- **検索結果そのものが0件** → ブロック / HTML構造変更の疑い。3回連続で `blocked` として自動停止し、画面に警告
- **結果はあるが新規企業が0件** → キーワードの掘り尽くし。5回連続で `exhausted` として停止（障害ではない）

一緒にすると、ただの枯渇を障害として報告し続けることになる。停止したソースは自動再開しない。

スコープ外の実装を思いついたら、**実装せずに提案として報告する**。

### 一括送信の入力: テンプレート or 直接入力

一括送信は **テンプレートモード** と **直接入力モード** の2系統を持つ。

- **テンプレートモード**: 従来どおり `templates` の文面を使う
- **直接入力モード**: 件名・本文を自由に入力。生成済みメールの引用も可能

生成済みメールを引用する場合、その本文は1社の分析結果から書かれている。
他社固有の事実（hook / recent_topics / activities 等）が含まれるため、
**そのまま複数社に配信してはならない**。該当箇所を `{{AI:指示}}` ゾーンや
差し込み変数（`lib/variables.ts`）に置き換えてから送信する。UIにもその旨を注記してある。

> 2026-07-19 にテンプレ専用にしたが、2026-07-20 にユーザーの明示的要求で直接入力モードを追加。
> `{{AI:指示}}` ゾーンの導入により、宛先ごとのカスタマイズが構造的に可能になったため。

### アクセス保護

`APP_PASSWORD` 環境変数で全ページ・全APIを保護する（`proxy.ts`）。
**未設定だと誰でも送信APIを叩ける**ので、本番では必ず設定する。設定画面に警告が出る。

### データ収集の方針（重要）

企業リストの入口は次の4本:

1. 商用企業DBの CSV / Excel 取込（F8）
2. 検索エンジン経由のキーワード検索（既存 keyword-search。Serper / DuckDuckGo のインデックスを見るもので、媒体サーバーを直接叩かない）
3. **Wantedly 新着の直接取得**（`lib/wantedly-scraper.ts`）。公開一覧ページのみ。ログイン必須領域やアンチボット回避はしない。リクエスト間隔は数秒＋ランダム化、同時実行1。取得0件が続いたら自動停止
4. 手動追加

相手企業**自身の**HPクロールは媒体規約と無関係なので継続する。

> Updated by 2026-07-20 ユーザー判断 — Wantedlyの直接スクレイピングをリスク承知の上で追加。規約違反のリスクはユーザーが受容。

## 絶対に守る制約

1. **`send_log` は不変**。送信後に削除・更新しない。監査証跡であり二重送信判定の正
2. **送信APIを叩く前に「送信中」状態を永続化する**。逆順にするとクラッシュ再実行で二重送信が起きる
3. **`suppressions` チェックは無効化できない固定仕様**。配信停止対応は特定電子メール法上の法的義務
4. **未解決の `{{変数}}` が残っていたら送信をブロック**。目視前提にしない（KB: `prompt-rebuild-field-dropout`・再発2回）
5. **エラーを握り潰さない**。送信失敗・API エラー・検証不合格は必ず画面に理由を出す
6. **OAuthトークンは暗号化保存**（鍵は環境変数）。トークン・APIキーを画面/ログに出さない。外部APIのエラー詳細をUIに露出しない
7. **相手企業のHPは信頼できない外部入力**。HP内の指示文にAIが従わないよう、分析プロンプトはデリミタで囲い、生成段には分析JSONのみ渡す（生HTMLを渡さない）
8. **URL入力は http/https のみ許可**。プライベートIP・localhost へのリクエストを禁止（SSRF対策。既存 `lib/ssrf.ts`）

## コーディング規約

- **Immutability 優先**: 既存オブジェクトを mutate せず、新しいオブジェクトを返す
- **サイズ**: 関数 <50行 / ファイル 200〜400行目安（上限800行）/ ネスト ≤4（超えたら早期return・分割）
- **命名**: camelCase（変数・関数）/ PascalCase（型・コンポーネント）/ UPPER_SNAKE_CASE（定数）/ boolean は `is` `has` `should` `can` 接頭 / カスタムフックは `use` 接頭
- マジックナンバー・ハードコード値は名前付き定数か設定に出す
- **境界での入力検証必須**（外部データ・APIレスポンス・ユーザー入力を信用しない）
- production コードに `console.log` を残さない
- 既存のコードスタイル・ファイル構成に合わせる。新しい流儀を持ち込まない

