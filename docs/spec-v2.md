## 2. v2.0 の目的とユースケース

### 目的

「商材を登録すると、それに合う企業リストが手に入り、1社ごとにパーソナライズされたメールが一括生成され、一覧で確認してチェックした分だけ実際に送信される」ところまでを1つのツールで完結させる。

### 代表ユースケース

**商材は固定しない（2026-07-18 確定）**。services に複数商材を登録し、キャンペーンごとに「商材 × 対象リスト × テンプレ × 送信者」を組み替えて使う汎用営業基盤とする。

- **UC1（例: きっかけインターン営業）**: 採用活動中の企業をリストアップ → 各社のHPを分析 → 「固定の書き出し + 各社に即したAI文面」を一括生成 → 一覧レビュー → チェックした分を担当者名義の Gmail から送信
- **UC2**: 営業リストCSV（企業DBサービス・Eight等から出力）を取り込み、担当者名既知の相手へ一斉パーソナライズ営業
- **UC3（先行設計のみ）**: 相手企業向けに生成した個社LPのURLをメールに添えて営業する（LP生成は別パイプライン。本ツールは URL の差し込みと送信を担う）。SpotRight・LP制作営業など商材が変わっても同じ仕組みで回せる

### v2.0 でやらないこと

- 完全自動送信（レビューなし送信）— 実装はするが機能フラグで無効。昇格条件は §8
- **求人媒体（Wantedly 等）の直接スクレイピング** — 調査の結果、確認できた10媒体すべてが営業目的利用・複製を規約で禁止。**Wantedly は「手動による方法を問わず」「AI またはエージェント型 AI」による転記まで明文で禁止**しており、目視転記でも規約違反になる（[research/media-tos.md](research/media-tos.md)、独立検証で原文確認済み）
- LP サイトの自動生成そのもの（別パイプライン。本ツールは URL を受け取るだけ）
- 開封率トラッキング（トラッキングピクセルは配信性を悪化させるため見送り。返信検知で代替）
- 問い合わせフォームへの自動入力・自動送信（引き続き対象外）

**💬 ざっくり**: 「求人サイトをロボットで読んで企業リストを作る」は、調べた結果どのサイトも規約で禁止しており、Wantedly に至っては“人間が目で見て書き写す”ことまで禁止と明記していました。なのでこのやり方自体を捨て、次の F1 の方法に切り替えます。

## 4. 機能要件

### F1: 企業リスト収集（確定案）

**採用する構成 = 二段構え**（[research/media-sources.md](research/media-sources.md) の推奨構成を採用）:

| 経路 | 内容 | 位置づけ |
|------|------|---------|
| **主軸（8〜9割）** | 商用企業DBサービスの「採用活動中」絞り込み機能で企業リストを抽出し CSV で本ツールに投入。候補: **Musubu**（求人媒体名で絞込可・1件約10円）/ **BIZMAPS**（「今年求人を行っている企業」タグ・月額約2.5〜5万円）/ **SalesNow Lite**（1件55円・契約不要の都度課金） | 事業者が公式機能として提供するデータのため規約・法的に最も安全。**ただし「インターン募集中」で絞れるかは各社未検証 → 契約前に無料トライアルで実地確認する（§11-U3）** |
| **補完（1〜2割）** | 手動発見（Google検索・プレスリリース・合同説明会の参加企業一覧等）で企業名を拾い、画面から手動登録 | 法的安全度が最も高い。DBに載らない企業を補完 |

**ツール側の機能**:

- `companies` への登録: CSV一括（F8と同じ取込口）/ 手動1件追加。同一ドメインは重複登録しない。**収集ソースを必ず記録**（後で「この経路は反応が良い」を分析するため）
- **採用シグナル検出**: 登録された各社の**公式HP**を既存 crawl.ts でクロールし、「採用」「インターン」「recruit」ページの有無・URLを自動判定して一覧に表示する。相手企業自身のHPを見る行為は媒体規約と無関係で、法的にクリーン
- 連絡先抽出（F2）・相性スコアリング（F3）はこのクロール結果に相乗りして同時実行

**不採用とした方式**（根拠は research/ 参照）:

- 媒体の直接スクレイピング・AI/目視転記 → 全媒体で規約違反（上記）
- 求人ボックス公式API → 用途が「自社サイトへの求人表示」に限定され営業リスト化は範囲外。使うなら運営へ書面許諾が必要（見送り）
- ハローワークの求人情報提供API → 職業紹介事業の許可・届出保持者限定で利用不可
- Indeed API → 求人検索用APIは2023年に廃止済み。現行APIは求人を「送り込む」側専用

**💬 ざっくり**: 「採用中の企業リストを売ってくれる正規のデータベースサービスからリストを買い、足りない分は人が手で探して足す」方式にします。ロボットで求人サイトを読む方式は全滅（全部規約違反）でした。どのDBサービスが「インターン募集中」まで絞れるかは無料お試しで確かめてから契約します。

### F2: 連絡先解決（メールアドレス・担当者名）

**メールアドレス**: 既存 crawl.ts の抽出ロジックを流用（mailto / 正規表現 / 難読化解除、優先順位: 採用・営業窓口系 > info@ > その他）。企業HPで公表されている事業者アドレスであること = 特電法の例外要件（§7-1）の基礎になるため、**取得元URLを記録する**。

**担当者名の解決チェーン**（上から順に採用）:

1. `contacts` に担当者名がある（商用DB・名刺由来など、**提供元が適法に保有するデータ**）→ **「◯◯株式会社 △△様」**
2. HP 分析で代表者名・採用責任者名が判明 → **「◯◯株式会社 代表取締役 △△様」等**（v1.1 §8 踏襲）
3. どちらも無し → **「◯◯株式会社 採用ご担当者様」**（キャンペーン単位で文言設定可、デフォルトは商材に応じ「採用ご担当者様」/「ご担当者様」）

**個人情報の取扱いルール**（[research/legal.md](research/legal.md) 反映）:

- 経路2（HPから取得した個人名）は**宛名への利用と分析結果の表示にとどめ、個人名を軸にした検索・蓄積・エクスポート機能は作らない**。HP掲載の個人情報を一方的に収集して営業利用することの適法性（利用目的の通知・公表義務の要否）は、名刺交換のケース（個人情報保護委員会FAQで適法確認済み）と異なりグレーであり、**要弁護士確認**（§11-L1）
- 自社プライバシーポリシーに利用目的（営業活動のための連絡先情報の利用）を公表しておく

**絶対ルール**:

- 宛名（企業名 + 担当者名 or フォールバック）が解決できないメールは**送信不可**（生成はできるが承認ボタンが無効化）
- 企業名は正式名称・(株)等の略記禁止（v1.1 §8 踏襲）

### F3: 商材マッチング

- 商材（Service）の `target` を使い、`companies` の各社に**相性スコア（high/medium/low + 理由）**を付与。既存の分析パイプライン（`lib/analyze.ts` の compatibility 判定）を流用
- キャンペーン作成時に商材を選ぶと、**相性スコア順に対象企業を提案**。low はデフォルト除外（手動で含めることは可能）
- スコアリングは1社ごとに HP クロール + AI 分析が走るため、**リスト取込時に非同期バッチで実行**し結果をキャッシュ（キャンペーン作成時には計算済み）

**💬 ざっくり**: 商材を登録しておくと、集めた企業リストに「この会社は相性◎/○/△」の印が自動で付き、◎の会社から順に営業をかけられます。

### F4: ハイブリッド文面（固定リード + AI続き）

templates テーブルを拡張し、テンプレートを3パートで構成する:

| パート | 内容 | AI の扱い |
|--------|------|----------|
| `fixed_part` | ユーザーが書いた冒頭文（商材の言いたいことをそのまま使う部分）。差し込み変数使用可 | **一字一句変更禁止**（変数解決のみ） |
| `ai_brief` | 続きを書かせる指示（例:「この後、相手企業のHPの内容に触れて商談依頼で締める」） | AI がこの指示 + 企業分析JSON で続きを生成 |
| モード | `full_ai`（従来通り全文AI）/ `hybrid`（固定+AI）/ `fixed_only`（差し込みのみ・AIなし） | キャンペーン単位で選択 |

**差し込み変数**（fixed_part / 件名 / ai_brief で使用可）:

```
{{company_name}}   企業正式名称
{{person_name}}    担当者名（F2のフォールバックチェーンで解決）
{{sender_name}}    送信者（人格）名
{{service_name}}   商材名
{{lp_url}}         個社LPのURL（F9。未設定なら生成時にエラー）
```

**品質チェックの拡張**（既存 `lib/quality-check.ts` に追加）:

- [ ] 未解決の `{{...}}` が本文・件名に残っていない（**最重要。残っていたら承認不可**）
- [ ] hybrid モードで fixed_part が改変されていない（完全一致検証）
- [ ] 宛名の企業名が宛先メールアドレスのドメイン/企業と整合（**誤差し込み=最悪の営業事故**の機械検知）
- [ ] 特電法の必須表示事項（§7-1）が署名に含まれている
- 既存チェック（文字数・フック言及・禁止表現・ハルシネーション）は全モードで維持

**💬 ざっくり**: 「最初の3行はこの文章をそのまま使って、続きはAIが相手に合わせて書く」ができるようになります。名前や会社名の差し込みミス、法律で必須の表示漏れは機械が検出して、直るまで送信ボタンが押せません。

### F5: 一括生成

- キャンペーン作成 → 対象企業/連絡先を確定 → 「生成開始」で全件をキュー処理
- 並列度 3〜5（Claude API のレート制限内）。進捗表示（N/M件 完了・失敗数）
- 失敗（クロール不可・AI エラー）は自動1回リトライ → それでも失敗なら「failed」として一覧に残す（**silent skip 禁止**）
- 生成済みアイテムは途中からでもレビュー可能（全件完了を待たなくてよい）
- コスト目安: 分析+生成 = 1通あたり数円〜十数円 → 200通/日 で **数百円〜3,000円/日** 程度

### F6: 実送信（Gmail API・送信元選択）— 調査確定版

**アーキテクチャ**（[research/gmail.md](research/gmail.md)、重要主張は公式一次情報で検証済み）:

| 項目 | 確定内容 |
|------|---------|
| 前提 | **Google Workspace 必須**（§11-U2 で契約有無を確認）。無料Gmailは OAuth テストモードのリフレッシュトークンが**7日で失効**し業務運用不能。Workspace なら送信上限も 500→2,000通/日 に上がる |
| OAuth アプリ | GCP プロジェクトを自社 Workspace 組織に置き、同意画面を **Internal（社内限定）** に設定。**Internal なら restricted scope でも Google の審査（CASA等）は不要**（公式3ドキュメントの突き合わせで確認済み） |
| スコープ | **`gmail.modify` 1本**（送信 + 返信検知・バウンス検知の受信箱読み取りをカバー） |
| 送信者方式 | **担当者ごとに本人の Workspace アカウントを個別 OAuth 連携**（選択肢A）。本人の「送信済み」に残り、返信も本人に届く。send-as エイリアス方式は不採用（**エイリアスは何人分作っても送信上限が本体と合算**＝公式確認済み）。担当者が10名規模になったらドメイン全体委任（選択肢C）へ移行検討 |
| 実装 | `googleapis` + Nodemailer `MailComposer` で RFC822 生成（**日本語件名・表示名の MIME エンコードは自動**）→ base64url → `users.messages.send`。下書きモードは `users.drafts.create` |
| トークン | 担当者ごとの refresh token を暗号化してDB保存（鍵は環境変数）。`tokens` イベントで更新を永続化。**`invalid_grant` は自動リトライせず**「要再認証」状態にして本人に再ログイン導線を出す |
| 返信検知 | 送信時の `threadId`・`Message-ID` を保存 → 定期ポーリングでスレッド新着を確認 → `send_status` を自動で `replied` に |
| バウンス検知 | `from:mailer-daemon` 検索で DSN をパース → 該当宛先を `suppressions` に自動登録 + `send_status` 更新 |

**送信制御**:

| 項目 | 仕様 |
|------|------|
| 日次上限 | 送信者ごとに設定可能。**初期値 50通/日から開始し段階的に増量**（ウォームアップ。業界プラクティス）。Workspace の公式上限は 2,000通/日（ローリング24時間）だが、律速は迷惑メール判定であり上限まで使わない |
| 送信間隔 | ランダム間隔（初期値 30〜90秒）で逐次送信。一斉バーストしない |
| 二重送信防止 | ①送信直前に `send_log` を照合し、同一メアドへ過去N日以内（デフォルト90日）の送信があればブロック ②`campaign_items` はユニーク制約（campaign_id × 宛先メアド） ③送信APIコール前に「送信中」状態を先に永続化し、クラッシュ再実行でも再送しない |
| 抑止リスト | 送信直前に `suppressions` を照合しブロック |
| kill switch | 送信中いつでも「中断」可能。中断時点までの送信分はログに確定 |
| 失敗処理 | API エラーは message_id 未発行を確認してから1回リトライ。失敗は failed として一覧表示 |
| 複数アカウント規律 | **1人の送信上限を回避する目的での複数アカウント使い回しは Gmail Program Policies（Circumvention 条項）違反**。担当者=実在アカウントの1対1を厳守 |

**送信健全性モニタと自動停止**:

- 送信ドメインを **Google Postmaster Tools** に登録し、迷惑メール報告率を定期確認（Google 要件: **0.3%未満**厳守）
- ツール側で**バウンス率を常時監視**: 直近送信のバウンス率が閾値（初期値 5%）超過、または連続バウンス N件（初期値 5件）で**キャンペーンを自動一時停止**して警告表示。気づかないうちの凍結を防ぐ
- ステータス自動化: 送信成功 → `sent`、返信検知 → `replied`、バウンス → `bounced`（G6 解消）

**💬 ざっくり**: 金谷さん・重南さんそれぞれの Gmail をツールに接続し、「この営業は重南さん名義で」と選んで送れます。送ったメールは本人の送信済みに残り、返信も本人に届きます。いきなり200通は送らず50通から徐々に増やし、エラー（宛先不明で跳ね返る等）が増えたら**ツールが自動でブレーキをかける**仕組みを入れます。会社で有料版Google（Workspace）を使っていることが前提条件です。

### F7: レビューキュー（半自動送信）— 決定済み方式

キャンペーン画面 = 生成結果の一覧テーブル:

| 列 | 内容 |
|----|------|
| ☑ | 送信対象チェック（品質チェック不合格・宛名未解決の行はチェック不可・理由表示） |
| 宛先 | メールアドレス + 会社名 + **解決された宛名**（実名 or フォールバックが一目で分かる色分け） |
| 件名/本文 | プレビュー（行クリックで全文・インライン編集・再生成） |
| 相性 | high/medium/low |
| 品質 | 合格/警告（理由をツールチップ表示） |
| 状態 | 未生成/生成中/生成済/承認済/送信中/送信済/失敗/スキップ |

**送信フロー**: チェック → 「選択したN通を送信」→ **最終確認モーダル（件数・送信元アドレス・日次残り枠を表示）** → 送信実行 → 行ごとに送信結果が反映。

**テストモード**（設定でON/OFF）: ON の間は**全宛先が自分たちのテスト用アドレスに強制上書き**され、画面に赤帯で「テストモード」を常時表示。

**運用工数の目安**: レビュー1通30秒とすると 200通/日 ≈ **100分/日**。2〜3名で分担する前提。回らない場合は「品質チェック合格 + 相性 high のみ全文レビュー、他はサンプリング」等の運用緩和を Stage 2 の実績を見て検討（勝手に緩和しない）。

### F8: 汎用CSV/Excel取込（⚠️ Eight専用連携から再定義）

> **調査による重大な発見**: Eight 利用規約 第12条には **19号「Eightサービスを用いて商用のサービスに利用する行為」・20号「Eightサイトにアップロードして登録した名刺の情報等を他のシステムに利用する行為」** の禁止が明文である（[research/critique.md](research/critique.md) で規約原文を直接確認）。**Eight からエクスポートした名刺CSVを本ツールに取り込んで一斉送信する行為は、20号にそのまま該当する可能性が高い。** また個人向け Eight の CSV エクスポート自体が有料（プレミアム）限定で、公開APIは存在しない。

**v2.0 での扱い（2026-07-18 ユーザー判断: リスク承知で Eight データを利用する）**:

- 機能は **「汎用の営業リストCSV/Excel取込」** として実装する（F1 の商用DB出力CSV・Eight エクスポートCSV の両方が同じ入口を通る）
- 取込: CSV（UTF-8 / Shift_JIS 自動判定）+ **Excel（.xlsx）対応**（G4 解消）。列マッピング UI（会社名・氏名・部署・役職・メールアドレス等の自動マッピング + 手動修正）。**Eight エクスポート形式の列プリセット**を同梱
- 取込時処理: メアド形式検証 / 重複排除（メアド基準）/ `suppressions` 照合 / `contacts` へ登録（source 必須記録。Eight 由来は `source=eight` で追跡可能にする）
- API 連携は行わない（個人向け Eight に公開APIは存在しない）。既存UIの Eight API 前提の実装と `eight_api_key` 設定は削除し、CSV 取込に一本化
- **リスクの記録**: Eight 利用規約 第12条19号・20号は名刺情報の商用利用・他システム利用を明文で禁止しており、エクスポートデータの本ツール投入は規約違反と判断される可能性が高い。ペナルティは Eight アカウントの停止等（法令違反ではなく Eight との契約上の問題）。**このリスクを承知の上で利用する判断をユーザーが行った（2026-07-18）**。弁護士確認（§11-L2）は引き続き推奨
- 名刺データ自体の個人情報保護法上の扱いは、名刺交換で直接取得したものなら適法（個人情報保護委員会FAQ確認済み）

**💬 ざっくり**: Eightの規約には“名刺情報を他のシステムで使ってはいけない”と書かれており、使うと（法律違反ではなく）Eightのアカウント停止などのリスクがあります。それを承知で使うというご判断だったので、Eight のCSVをそのまま読み込めるプリセット付きで機能を作ります。この判断の経緯は要件書に記録として残します。

### F9: 個社LP営業の受け皿（先行設計 + LP制作ルール）

> 構想: 相手企業向けの LP を生成し、メールに添えて「もう作ってみました」型の営業を仕掛ける。**LP の生成・ホスティングは本ツールの外**。本ツールは「LP と宛先の紐付け」「メールへの組み込み」「送信」を担う。

**メール側の仕様**:

- `contacts` / `companies` に `lp_url` を持ち、CSV取込時 or 画面から紐付け
- 文面には `{{lp_url}}` で差し込み。**lp_url 未設定の宛先が LP キャンペーンに含まれる場合は生成時にエラー**
- **添付はしない（確定）**: HTML添付は法人側セキュリティ（Google Workspace 管理者設定）で機械ブロックされ得る + フィッシングの典型手口として警戒される。PDF添付も初回コールドメールでは非推奨が業界コンセンサス
- **リンクは本文に1本だけ・短縮URL禁止・LP専用ドメインへの直リンク**。リンク先が何かを本文で明示（Google 公式ガイドライン準拠）
- **Gmail は本文URLを自動でOGPカード表示しない**（LINE/Slack と違う）。「サムネイルで見せる」前提は成立しないため、見せたい場合は本文テキストでLPの内容を1行説明する

**LP制作側に課すルール**（別パイプラインへの要求仕様。[research/lp.md](research/lp.md) 反映）:

1. **相手企業の実写真・実ロゴ・サイト文章を使わない**。プレースホルダー + テキストで「この構成・デザインで作れます」という提案体裁にする。理由: 著作権法の**送信可能化権はサーバーにアップした時点で成立**し、「URLは本人にしか送っていない」「noindex にした」は抗弁にならない可能性が高い。ロゴ使用は誤認時に不正競争防止法リスク
2. ホスティングは**コーポレートドメインと分離した LP 専用の独自ドメイン**。`*.vercel.app` 等の無料PaaSサブドメインは実際にフィッシング悪用実績があり受信側に警戒されるため使わない
3. 全LPに **noindex**（大量の個社LPは Google の scaled content abuse ポリシーに触れ、本体ドメインのSEO評価を毀損し得るため）
4. LP上で有料申込へ誘導する場合は特定商取引法の表示が必要になる可能性 → 商材確定後に確認（§11-L3）
5. 効果の参考値: パーソナライズLP で返信率約2倍、パーソナライズ動画で2〜4倍（いずれもベンダー自社データ・未検証）。実測で判断する

**💬 ざっくり**: 「御社向けにサイトを作ってみました → リンクはこちら」方式で行きます（ファイル添付は迷惑メール直行コースなので却下）。ただし法律上いちばん危ないのは“相手の会社の写真やロゴを勝手に使うこと”で、「本人にしかURLを送ってないから大丈夫」は通用しません。LPはダミー画像+文章で「こういうものを作れます」という見せ方にします。

### F10: オプトアウト・抑止リスト

- `suppressions` テーブル: メールアドレス / ドメイン単位。理由 = 配信停止依頼・バウンス（F6から自動登録）・「営業お断り」記載検出・手動登録
- **「営業お断り」検出は法的要件**: 公表アドレスに送信拒否文言が併記されている場合、特電法の例外が適用されず送信は違法になる（§7-1）。crawl 時にアドレス周辺の拒否文言を検出したら suppressions に自動登録し、確認画面に理由を表示
- 配信停止依頼が来たら（当面は人間が受信を確認して）画面から登録 → 以降すべてのキャンペーンで送信ブロック

### F11: 返信の仕分け + 返信下書き

送信して終わりにせず、**返り**を捌くところまで自動化する。200通/日で返信率5%なら1日10件の対応が発生し、ここが次のボトルネックになる。

- Gmail のスレッド監視（F6 の返信検知）で返信を取得 → **AI が自動分類**:
  | 分類 | 挙動 |
  |------|------|
  | 興味あり・詳細希望 | **返信下書きを自動生成**して通知。人間が確認して送信 |
  | 日程調整したい | 返信下書き + 日程調整リンク（F14）を含めた案を生成 |
  | 断り・見送り | `send_status = rejected`。**suppressions に自動登録**（再アプローチ事故の防止） |
  | 不在・後日連絡 | フォローアップを指定日まで自動延期（F12 と連動） |
  | 自動返信（不在通知等） | シーケンス継続。返信としてカウントしない |
  | 配信停止依頼 | **suppressions に即時登録**（法令要件・§7-1） |
- 返信本文は**外部からの信頼できない入力**として扱う（v1.1 §7.2 のプロンプト注入対策を適用。返信文中の指示に AI が従わないこと）
- 分類結果は必ず人間が確認できる一覧に出す。**AI の分類だけで送信・停止を確定させない**（配信停止依頼と断りは安全側に倒して即時抑止する）

**💬 ざっくり**: 返ってきたメールを「興味あり/断り/日程調整したい」などに自動で仕分けて、脈のある相手には返事の下書きまで用意しておきます。断られた相手は二度と送らないリストに自動で入ります。

### F12: フォローアップシーケンス（調査確定版）

> **数値の信頼性について**: 「何日後に送るべきか」の相場観の大半は**コールドメール配信ベンダーの自社データ**であり、独立検証可能な学術研究・公的統計は存在しない。業界通説の「3-7-7の法則」も原典を特定できなかった。**以下は初期値であり、運用開始後は自社の実返信率・苦情率で検証して上書きする前提**。全文は [research/SPEC-cadence.md](research/SPEC-cadence.md)。

**推奨シーケンス: 初回 + フォローアップ2通（合計3通）・営業日ベース**

| 通数 | タイミング | 件名の扱い | 内容の狙い |
|------|-----------|-----------|-----------|
| 1通目 | Day 0 | 新規件名 | パーソナライズした提案。**日程調整リンクは入れない**（F14 参照） |
| 2通目 | 初回から **3営業日後** | 同スレッド継続（Re:） | 軽いリマインド + 新情報を1点追加。反応（開封・クリック）があればここで初めて日程調整リンクを投入 |
| 3通目（最終） | 初回から通算 **8〜9営業日後**（2通目から5〜6営業日後） | 同スレッド継続。ただし**開封ゼロが続く宛先は新件名に切替** | ソフトクローズ（「タイミングでなければ一旦クローズします」） |

**なぜ3通か（欧米の定石=4〜7通から意図的に減らしている）**:

- 欧米ベンダーが推す「初回+3〜5通」の根拠数値（Woodpecker「返信率8.3% vs 4.1%」等）は、独立検証の結果**自社データへの帰属が裏付け不十分**（記事内で他社データと混在・出典注記なし）と判明。通数を積み増す根拠として採用できない
- 「4通超で苦情リスク3倍」も測定方法・母数が非開示の業界通説
- **唯一検証可能なハード上限は Gmail 公式の迷惑メール報告率 0.3%**（超えると緩和措置の対象外・理想は0.1%未満）。フォローアップは総送信量を押し上げ苦情率に直結する
- 日本では「追いすぎ」への忌避感が指摘される（実務家の所感であり実証データではない）
- → **保守的に3通から始め、苦情率を見ながらA/Bで増減を判断する**

**開封ゼロの診断的活用**: 同スレッドで開封が全く付かない = 初回が迷惑メールフォルダに落ちているシグナル。3通目で新件名に切り替え、それでも反応が無ければ F29（受信トレイ到達テスト）で到達性を疑う。

**送信曜日・時間帯のデフォルト**（F20 と連動）: **火〜木曜 / 10時台または13時台**。※日本のBtoBコールドメールに特化した独立検証データは存在せず、メルマガ・テレアポという隣接チャネルからの合成仮説。**必ず自社データでA/Bテストして上書きする前提**の初期値。

**停止条件**:

| 条件 | 対応 | 自動/手動 |
|------|------|----------|
| 返信あり（内容問わず） | 即停止 | 自動 |
| ハードバウンス（5.x.x） | 即停止 + suppressions 登録 | 自動 |
| ソフトバウンス連続 or 7日超 | 一時停止・手動確認 | 半自動 |
| 不在通知（自動返信） | **返信としてカウントせず、自動再開もさせない**。人の確認を経て再開判断 | 手動 |
| 配信停止依頼 | 恒久停止。**特電法上の法的義務のため、この停止条件だけは無効化できない固定仕様** | 自動 |
| 全通で開封ゼロ | 個別停止 + 同一ドメインの到達性を調査 | 手動 |
| 迷惑メール報告率が閾値接近 | **キャンペーン全体を一時停止**（アラート閾値 0.2% = Gmail危険域0.3%の手前） | 自動アラート + 手動判断 |

**ユーザーが変更できる設定項目**: 総通数（1〜7通・上限超は警告）/ 各間隔の営業日数（1〜15）/ 営業日・暦日の切替 / 送信曜日 / 送信時間帯 / 日程調整リンクの投入タイミング / 件名の扱い（常にRe: / 常に新件名 / 自動切替）/ 不在通知時の自動再開（デフォルトOFF）/ 苦情率アラート閾値（0.05〜0.3%）/ ソフトバウンス保留期間

**実装メモ**: 既存の followup 生成API（`app/api/prospects/[id]/followup/route.ts`）を土台に拡張できる。フォローアップも F7 のレビューキューを通す（Stage 2 の間は人間承認が必須）。**停止漏れ = 断った相手への追撃メール**は重大事故なので、送信直前に停止条件を必ず再チェックする。

**💬 ざっくり**: 「3日後→さらに5営業日後」の合計3通で止める設計にしました。海外のツールは「7〜8通しつこく追え」と言いますが、その根拠になっている数字を検証したら出所が怪しく、日本では嫌われるリスクもあるので保守的にしています。数字は全部あとから変えられるようにして、実際の反応を見て調整します。

### F13: 効果測定ダッシュボード

「何が効いたか」が分からないまま送り続ける状態を作らない。

- **軸別の返信率・商談化率**: テンプレ別 / 商材別 / 送信者（人格）別 / リスト仕入れ元別 / 相性スコア別 / 何通目か別 / 送信曜日・時間帯別
- **件名A/Bテスト**: 同一キャンペーン内で件名を2案に分けて自動割当 → 開封の代替指標として**返信率**で比較（開封率トラッキングは §3 の通り不採用）
- ファネル表示: 送信 → 返信 → 商談化（F14 の予約成立）→ 受注（手動入力）
- 送信健全性（バウンス率・迷惑メール報告率）も同画面に集約

**💬 ざっくり**: 「どの文面・誰の名前・どのリストが一番反応が良かったか」を数字で出します。当たりパターンが見えたらそこに寄せられます。

### F14: 日程調整リンク差し込み（調査確定版）

現在 **Calendly** を利用中（2026-07-18 確認）。**複数ツールを選択可能**にする方針は維持する。

**⚠️ 1通目には入れない（確定）**

- CTAの種類別の返信率（Gangly社・自社500通データ・独立検証不可）: 関心を問う質問型 **11.4%** > 資料送付提案 8.2% > 直接会議依頼 5.0% > **カレンダーリンク 3.1%** > 複数候補提示 2.7%。**「依頼の大きさに反比例して返信率が下がる」が最も再現性の高い知見**
- 日本の一次データ（ミクステンド/TimeRex・マクロミル委託 n=309）では **92.6%が「URL送付は失礼ではない」**と回答。ただし**失礼と感じた人の理由の最多（56.5%）が「承諾前なのに会う前提の姿勢」** — これは英語圏の「いきなり会議依頼は返信率が下がる」と**同じ方向を向いている**
- → **1通目は低摩擦のCTA（関心を問う質問）で締め、リンクは2通目以降・相手の反応を見てから投入**（F12 のシーケンスと連動）
- 年配層・目上の相手には「ご都合が合わなければお知らせください」等の一言を添える文面ルールをテンプレに組み込む

**ツール構成（2026-07-18 確定）: Calendly 継続。有料プラン契約済みのため Webhook が利用可能**

- Calendly の Webhook は Standard 以上の有料プランで提供される。**既に課金済みのため、予約完了の自動検知をそのまま実装できる**（TimeRex 等への切替は不要）
- **予約完了 Webhook を受けて自動更新する**: `campaign_items.status` / `send_status = meeting` / F13 のファネル計測（送信→返信→**商談化**）/ F16 の企業ロック（予約済み = 即クローズ扱い）
- 招待者情報のプリフィル（相手の氏名・会社名を予約フォームに事前入力）も URL クエリパラメータで対応可能 → 入力の手間を減らして予約完了率を上げる
- **他ツールへの差し替え余地は残す**: `senders.booking_tool`（calendly / timerex / spir / google / other）+ `booking_url` の汎用設計とし、Webhook 受信部だけツール別に実装する。将来 TimeRex 等に替えても本体を書き換えずに済む

**配信到達性への配慮**

- `calendly.com` 等は**多数の送信者が共有する高トラフィックドメイン**であり、他者の悪用でドメイン評判が落ちると自社の送信履歴と無関係にリンク入りメールが不利になる。Mimecast 等のセキュリティゲートウェイが共有ドメインリンクを機械的にブロックした事例も報告されている
- 対策: **リンクは本文に1本まで**（F9 と同じ方針）/ 短縮URL不使用 / 可能なら自社ドメインのリダイレクトを噛ませる

**実装**: `senders` に `booking_tool` と `booking_url` を持ち、`{{booking_url}}` で差し込む。予約完了 Webhook を受けるエンドポイントを用意し、`campaign_items.status` と `send_status` を自動更新する。

**💬 ざっくり**: 1通目にはカレンダーのリンクを入れません。データを見ると「いきなり予約させようとする」ほど返信率が下がり、日本の調査でも嫌う人の理由の1位が「まだ会うと言っていないのに会う前提」でした。2通目以降、相手が反応してから出します。あと、Calendlyは無料プランだと「予約が入った」の自動検知ができないので、そこを自動化したいならTimeRexに替えるか有料化が要ります。

### F15: LP閲覧検知

個社LP（F9）を**相手が見たかどうか**を検知し、熱いうちにフォローする。

**実装方式は「宛先ごとの識別子付きURL + サーバーアクセスログ」に確定**（調査結果。全文は [research/lp-tracking.md](research/lp-tracking.md)）

| 方式 | 法規制上の扱い | 判定 |
|------|--------------|------|
| **(a) URL識別子 + サーバーログ**（`/lp/abc123`） | ブラウザ標準の HTTP リクエストのみで、外部送信規律が定義する「情報送信指令通信」に該当しない。Cookie不使用 | ✅ **採用** |
| (b) JS計測（滞在時間・スクロール率を自社APIへ送信） | 規律上の「外部」は**自社サーバーへの送信も含む**。滞在時間等は「サービス提供に真に必要な情報」とは言い難く、グレー | 見送り |
| (c) Google Analytics 等 | 明確な第三者への送信 = 外部送信規律の本丸。氏名紐付け設定をすると個人情報保護法31条にも踏み込む | ✕ 不採用 |

**法的整理（調査結果）**:

- **電気通信事業法の外部送信規律**: 個社向け営業LPは「自社の営業活動の手段として電気通信を用いているに過ぎない = **自己の需要**」であり、そもそも電気通信事業に該当せず規律の対象外となる可能性が高い（総務省パブコメの「自社商品のEC販売サイトは該当しない」という整理からの類推）。※個社限定LPを名指しした公式Q&Aは存在せず類推解釈 → §11-L4 で弁護士確認
- **個人情報保護法**: 氏名・会社を既に適法に保有している相手の閲覧ログを紐付ける行為は、第三者提供を伴わない**自社内処理**のため31条（個人関連情報の第三者提供規制）の対象外。適用されるのは通常の17条・18条 → **プライバシーポリシーに「取得情報を営業活動（送付資料・ウェブページの閲覧状況の把握を含む）に利用する」旨を明記・公表**しておけば足りる
- **Cookie同意バナー**: 方式(a)は Cookie を使わないため**そもそも不要**。日本の個人情報保護法にGDPR型の事前同意義務はない。日本語・日本企業向けのため GDPR 域外適用も通常想定しがたい

**メール開封トラッキングを採用しない根拠が補強された**:

- Apple Mail Privacy Protection により**全開封イベントの55〜60%が誤検知**（Litmus のライブ集計・月次更新・ベンダー自社データ）。開封率という指標自体がすでに信用できない
- Google 自身が「Googleは開封率を追跡していない」「低い開封率が必ずしも配信性の問題を示すわけではない」と公式に明言
- ピクセルは受信者に無通知で発火するため、**プライバシー侵害と感じた受信者の迷惑メール報告 → Gmail が実際に見ている0.3%指標の悪化**という経路でこそ危険
- → **LP側計測は「相手が自分でクリックして訪問した先」の計測であり、メール本文に手を加えないため送信ドメイン評判に一切影響しない**

**機能仕様**: 宛先ごとに一意のLP URLを発行 → 閲覧（初回・再訪・回数）を検知 → キャンペーン一覧に「閲覧済み」バッジ + LINE即時通知（F23）。「閲覧したが返信なし」の相手はフォローアップ文面を変える（F12 と連動）。※滞在時間はアクセスログのタイムスタンプ間隔から概算できる範囲にとどめる（精密に取ろうとすると方式(b)のグレーゾーンに入る）。

**💬 ざっくり**: 「宛先ごとに違うURLを渡して、あとはサーバーの来訪記録を見るだけ」という一番シンプルな方式にします。Cookieを使わないので同意バナーも不要、法律上のグレーゾーンも避けられ、実装も一番軽いです。「何秒見たか」を精密に取ろうと欲を出すと、途端に法律とコストの両方が重くなることが分かりました。

**💬 ざっくり**: 作ったLPを相手が開いたら分かるようにします。今まさに見ている会社を優先して追いかければ確度が上がります。メール自体には仕掛けを入れないので迷惑メール判定は悪化しません。

### F16: 担当バッティング防止

**同じ会社に社内の別の人が知らずに営業する事故**を防ぐ。二重送信防止（同一メールアドレス）とは別レイヤ。

- `companies` に**担当者（sender）と最終接触日**を持たせ、企業単位でロック
- キャンペーン作成時・送信直前に、**同一企業へ他の送信者が過去N日以内（デフォルト90日）に接触していないか**を照合 → 該当したら警告表示し、対象から除外 or 明示的に上書き承認
- 企業詳細に**接触履歴タイムライン**（誰がいつ何を送り、どう返ってきたか）を表示
- 会社の同定はドメイン基準 + 企業名の正規化（`info@` と `saiyo@` が同一社であることを認識できるように）

**💬 ざっくり**: 金谷さんと重南さんが同じ会社に別々に営業して「御社バラバラですね」と思われる事故を防ぎます。誰がいつ接触したかの履歴も会社ごとに見えます。

### F17: 電話・フォーム併走リスト

メールで反応が無い企業を**捨てずに別チャネルへ渡す**。

- クロール時に**電話番号・問い合わせフォームURL**も抽出して `companies` に保存（既存 crawl.ts の拡張。フォームURL は既に取得済み）
- シーケンス完走（F12）しても無反応の企業を「**メール枯れリスト**」として抽出し、架電リスト CSV として書き出し
- 架電結果（繋がった/不在/断り）を手入力で戻せる。断りは suppressions へ
- **フォーム自動送信は引き続き実装しない**（v1.1 からの方針。リンクを開くだけ）

**💬 ざっくり**: メールで反応が無かった会社の電話番号を一覧で出します。リストを買っているのに1チャネルで捨てるのはもったいないので、電話に回せるようにします。

### F18: 危険ワード・事実誤認の検知（送信前の最終ゲート）

**AIが相手企業について嘘を書くのが最悪の事故**。既存 `lib/quality-check.ts` を拡張する。

| 検知対象 | 内容 |
|---------|------|
| ハルシネーション | 本文中の固有名詞・数値・実績が**分析JSONに存在するか**を照合。存在しない事実に言及していたらブロック（v1.1 §8.3 の強化） |
| 断定・誇大表現 | 「No.1」「必ず」「絶対」「業界唯一」等 → 景品表示法リスク。警告 |
| 誤った敬語・二重敬語 | v1.1 §8.1 のルール違反 |
| 相手企業への誤解 | 分析の相性が low なのに「まさに御社にぴったり」等、分析結果と矛盾する主張 |
| 社名の表記ゆれ | 正式名称と不一致（(株)略記・旧社名など） |

- **ブロック（送信不可）**と**警告（人間が確認して押し切れる）**を分ける。ハルシネーションと社名不一致は**ブロック**
- 検知理由はレビューキュー（F7）の行に理由付きで表示する

**💬 ざっくり**: AIが「御社の◯◯という実績を拝見し」と勝手に事実を作って書いた場合、送信前に止めます。相手について嘘を書いたメールが飛ぶのが一番まずいので、ここは押し切れない仕様にします。

### F19: 業種別 言い換え辞書

同じ商材でも**相手の業種で刺さる言い方が変わる**のを資産として溜める。

- 商材 × 業種の組み合わせごとに「訴求フレーズ候補」を保持し、生成プロンプトに注入
- F13 の効果測定と接続し、**返信率の高かったフレーズを自動で昇格候補にする**（採用はユーザー判断。ツールが勝手に辞書を書き換えない）
- 初期は手動登録でよい。データが溜まってから自動サジェストへ

### F20: 送信予約・営業時間制御

- キャンペーンに**送信ウィンドウ**を設定（デフォルト: 平日 9:00〜18:00）。ウィンドウ外の送信はキューに留めて翌営業日に持ち越し
- 祝日カレンダー対応（日本の祝日に送らない設定）
- 送信間隔ランダム化（F6）と組み合わせて自然な送信パターンにする
- **具体的な推奨時間帯は §13 の調査結果で確定**

**💬 ざっくり**: 深夜2時や日曜日に営業メールが届かないよう、平日の日中にだけ配信します。

### F21: 既存顧客・取引先の除外

**すでに取引がある会社に「はじめまして」メールが飛ぶ**のは信用問題。F16（社内バッティング）とは別レイヤの恒久ブロック。

- 「除外リストCSV」を投入（顧客リスト・商談中リスト・パートナー企業・競合他社・自社グループ）
- 除外は**ドメイン単位 + 企業名の正規化マッチ**（`info@` と `sales@` の両方を1社として弾く）
- suppressions とは別テーブルにする（理由と運用が違うため。除外リストは定期的に最新の顧客リストで洗い替える運用）
- キャンペーン作成時に**除外件数を必ず表示**（「120社中18社が除外リストに該当」）

**💬 ざっくり**: すでにお付き合いのある会社に新規営業メールが飛ぶ事故を防ぎます。顧客リストを入れておけば自動で弾きます。

### F22: 添付資料の出し分け

- **初回メールには添付しない**（F9・調査で確定した方針）
- 返信が来た後の2通目以降、または「資料希望」と分類された返信（F11）に対して、services に登録済みの**商材PDFを添付**できる
- 添付可否はテンプレ側で制御（初回テンプレは添付欄そのものを無効化し、事故を構造的に防ぐ）

### F23: 週次サマリの LINE 通知

ダッシュボード（F13）は**見に行かないと見ない**。プッシュして初めて運用が回る。

- 毎週月曜（時刻設定可）に前週サマリを **LINE に自動投稿**: 送信数 / 返信数・返信率 / 商談化数 / バウンス率・健全性の警告 / 今週の推奨アクション（例「リスト残りわずか」「テンプレAの返信率が低下」）
- 加えて**即時通知**も同じ経路で送る: LP閲覧の検知（F15）、興味あり返信の着信（F11）、健全性の自動停止発火（F6）
- ⚠️ **実装上の注意**: 手軽な `LINE Notify` は**サービス終了済み**（2025年3月末で提供終了）のため使えない。**LINE Messaging API（push message）** で公式アカウントを作り、通知先ユーザー/グループに配信する構成になる。または LINE WORKS の Bot API（社内利用ならこちらも適合）。**着手前に最新の提供状況を再確認すること**
- 通知先・通知する種別（週次のみ / 即時も）はユーザーごとに設定可能に

**💬 ざっくり**: 毎週月曜に「先週こうでした、今週これをやりましょう」が LINE に届きます。LPを見られた時や脈のある返信が来た時もその場で LINE に飛ばせます。※以前よく使われた LINE Notify は終了しているので、公式アカウント方式で作る必要があります。

### F24: 休眠リストの掘り起こし

**「無反応 = 永久に脈なし」ではない**。タイミングが悪かっただけのケースが大半なので、買ったリストを資産として再利用する。

- シーケンス完走（F12）後も無反応の企業を「休眠」として保持し、**一定期間（デフォルト180日）経過で再アプローチ候補に自動復帰**
- 再アプローチ時は**必ず文面・切り口を変える**（前回と同じ文面の再送は禁止。F26 の類似度チェックを適用）
- **対象は「無反応」のみ。明示的に断られた企業（rejected）・配信停止依頼は永久に対象外**（suppressions が優先。ここを間違えると重大事故）
- 復帰した企業には「◯回目のアプローチ・前回◯年◯月」を一覧に表示

**💬 ざっくり**: 「前は反応が無かったけど今なら刺さるかも」という会社を半年後に自動でリストへ戻します。ただし一度はっきり断られた相手は絶対に戻しません。

### F25: リスト枯渇アラート

- 未アプローチの宛先残数を常時カウントし、**現在の送信ペースで何日分残っているか**を算出
- 閾値（デフォルト3日分）を切ったら F23 の LINE 通知で警告
- 「営業が止まる原因の1位はリスト切れに気づかないこと」への対策。F23 に相乗りするため実装は軽い

### F26: 文面の類似度検知

**Gmail は「似た文面が大量に飛んでいる」ことを学習してスパム判定する。** AI生成でも骨格が同じになりがちで、しかも人間は気づけない。

- 同一キャンペーン内・および直近送信分と、生成文面の**類似度を機械的に測定**（本文の n-gram 一致率等）
- 閾値超過でレビューキューに警告表示 → 再生成を促す
- 固定リード文（F4 の `fixed_part`）は当然一致するため**除外して測定**する
- F24 の再アプローチ時は「前回送った文面との類似度」もチェック

**💬 ざっくり**: AIが書いても結局似たような文面ばかりになっていないかを機械が測ります。似すぎるとGmailに迷惑メール扱いされるためです。

### F27: 送信ドメインのブラックリスト監視

- cypherone.co.jp（および送信に使うドメイン・IP）が **Spamhaus 等の RBL（ブラックリスト）** に登録されていないかを定期チェック（無料の照会APIあり）
- 検知したら**即座に送信を停止**し LINE 通知（F23）
- **載ると営業メールだけでなく請求書等の通常業務メールも届かなくなる**ため、被害の大きさに対して監視コストが極めて低い

### F28: スマホ件名プレビュー

- **スマホの受信一覧では件名が15〜20文字程度しか表示されない**。開封は件名の頭で決まるため、切れた状態をレビューキュー上でプレビュー表示する
- PC幅・スマホ幅の両方で件名の見え方を確認できるようにする（v1.1 §4-⑤ の件名ルール「20〜35文字」の運用を実機基準に補正）

### F29: 受信トレイ到達テスト（シードテスト）

**「そもそも届いているのか」を知る手段が無い状態を作らない。** 返信ゼロの原因が①文面 ②リスト ③迷惑メールフォルダ行き のどれかを切り分ける。

- 自分たちが管理する**シードアドレス群**（Gmail / Outlook / Yahoo / 独自ドメイン等）を登録
- キャンペーン送信前・および定期的に、実際の文面をシードアドレスへ送信 → **受信トレイ / プロモーションタブ / 迷惑メールフォルダ のどこに入ったかを自動判定**（各アカウントを OAuth 接続して受信箱を検索。F6 の `gmail.modify` スコープで Gmail 分は実装可能）
- 迷惑メール判定された場合は**キャンペーン送信前に警告**し、文面・リンク・送信設定の見直しを促す
- ※③だった場合、文面をいくら改善しても永久に成果は出ない。**この機能が無いと改善のループが閉じない**

**💬 ざっくり**: 送ったメールが相手の「受信トレイ」に入ったのか「迷惑メール」に入ったのかを、自分宛てのテスト送信で先に確かめます。届いてすらいないのに文面を直し続ける、という一番もったいない状態を防ぎます。

### F30: メール認証の自己診断

- cypherone.co.jp（および送信ドメイン）の **SPF / DKIM / DMARC レコードを DNS から取得して判定**し、設定画面に ✅/⚠️ で表示
- 判定内容: レコードの存在 / 構文の妥当性 / SPF の include に Google が含まれるか / DKIM セレクタの有無 / DMARC ポリシー（none/quarantine/reject）
- §9 の前提インフラを「**設定したつもりで間違っている**」まま送り始める事故を防ぐ。DNS を引くだけなので実装は軽い
- 未設定・不備がある間は**送信開始をブロックする**（初回セットアップのゲートとして機能させる）

### F31: AI生成コストの見える化

- Claude API の**トークン使用量と概算金額**を日次・月次で表示（分析 / 生成 / 返信仕分け など用途別の内訳）
- 月次上限を設定し、超過が見込まれる場合は F23 の LINE 通知で警告
- 200通/日 × 1通あたり数円〜十数円 = **月数万円規模**になりうるため、気づかないうちに膨らむのを防ぐ

### F32: スマホからの承認

- レビューキュー（F7）を**モバイル幅で操作できる**ようにする（既存アプリにモバイル対応の下地あり: `app/nav-header.tsx` にボトムタブ実装済み）
- 1日あたり約100分と試算されるレビュー作業（§4-F7）を移動中・隙間時間に消化できるようにし、**運用が続くかどうか**を担保する
- スマホでは「全文確認 → 承認」に絞り、生成・設定変更は PC 前提でよい

## 7. 法令・コンプライアンス（調査確定版）

> 詳細と条文出典は [research/legal.md](research/legal.md)。罰則条文は e-Gov 法令APIの条文実体で検証済み。

1. **特定電子メール法**:
   - **B2B例外（第3条1項4号）**: Webサイトで自己のメールアドレスを公表している団体・営業を営む個人宛は事前同意なしで送信可。**ただしアドレスに「営業メールお断り」等の拒否文言が併記されていれば例外は不成立 = 送信禁止**（F10 で機械検出）
   - **表示義務（第4条）**: 全送信メールに①送信者の氏名/名称 ②住所 ③受信拒否を受け付ける旨 ④受信拒否の通知先（メールアドレス or URL）⑤苦情・問合せ先 を表示 → **署名テンプレートの必須項目とし、品質チェックで欠落を検出**（F4）
   - 罰則: 措置命令（第7条）違反で 1年以下の拘禁刑 or 100万円以下の罰金（第34条2号）、**法人は3,000万円以下の罰金**（第37条1項1号）
2. **個人情報保護法**: 名刺交換で取得した連絡先への営業メールは適法（個人情報保護委員会FAQ・法第21条4項4号）。**HP掲載の担当者名を一方的に収集して営業利用することの適法性はグレー** → F2 のルール（宛名利用のみ・蓄積機能なし）+ プライバシーポリシー公表 + 弁護士確認（§11-L1）
3. **Eight 規約**: 第12条19号・20号により名刺情報の他システム利用が禁止 → F8 の扱い + 弁護士確認（§11-L2）
4. **媒体規約**: 確認した10媒体すべてが営業目的利用を禁止 → F1 で媒体スクレイピング不採用（確定）
5. **Google 送信者ガイドライン（2024年2月〜）**: 5,000通/日未満でも全送信者に SPF または DKIM・PTR・TLS・RFC5322準拠・**迷惑メール報告率0.3%未満**が必須。DMARC（p=none 可）とリスト衛生は通数によらず設定推奨 → §9 の前提インフラ

## 8. 半自動 → 自動化の昇格ルール

| 段階 | send_mode | 内容 |
|------|-----------|------|
| Stage 0 | draft_only | Gmail の下書き作成のみ（送信APIは呼ばない）。接続・文面・差し込みの動作確認 |
| Stage 1 | テストモード × review_send | 宛先を自分たちに強制上書きして実送信。通しの検証 |
| Stage 2 | **review_send（v2.0 の標準）** | 一覧チェック → 承認分のみ実宛先へ送信 |
| Stage 3 | auto_send | 品質チェック合格分を自動送信（不合格は必ず人間レビューへ）。**実装はするが機能フラグで無効** |

**Stage 3 解禁の条件（案・§11-U6）**: Stage 2 で累計 100通以上送信し、①差し込み事故 0件 ②二重送信 0件 ③品質チェックの誤通過が直近 50通で 0件 ④迷惑メール報告率 0.3%未満・バウンス率 5%未満を維持。**解禁の判断はユーザーが行う**（ツールは勝手に昇格しない）。

**💬 ざっくり**: 「下書きだけ→自分宛てテスト→人が選んで送る」と段階を踏み、100通ノーミス+苦情ゼロを確認できたら、全自動モードを解禁するかをお二人が判断する設計です。

## 9. 前提インフラ・非機能要件

**前提インフラ（Phase A 着手前に完了させる。アプリ外の作業）**:

1. **Google Workspace**: **契約済み（2026-07-18 確認）**。送信担当者分のアカウントが Workspace 上にあることを確認
2. **送信ドメイン認証**: 送信ドメインに **SPF・DKIM・DMARC（p=none から）** を設定。未設定のまま送るとアプリが完成していても迷惑メール直行
3. **Google Postmaster Tools** に送信ドメインを登録（迷惑メール報告率の監視用）
4. 送信ドメイン方針: v2.0 は**本人名義=コーポレートドメインで送信開始**（「その人からのメール」という要件を優先）。ウォームアップ・レビューキュー・健全性監視でドメイン評判を守り、悪化の兆候が出たら営業専用ドメインへの分離を検討
5. LP 用の**別ドメイン取得**（F9 を使う場合）

**非機能**:

- **最優先は誤送信・二重送信の防止**（F6）。速度より安全
- 送信系の全操作に監査ログ（send_log は削除不可）
- OAuth トークンは暗号化保存・画面/ログに出さない。API エラー詳細を UI に漏らさない
- SSRF 対策（既存 `lib/ssrf.ts`）はリスト収集のクロールにも適用
- **稼働環境（2026-07-18 確定）: まお氏が保有する常時稼働のローカルPC上で動かす。** 具体的な構成・デプロイ手順・プロセス管理は実装時に蛯名氏とまお氏で調整するものとし、本要件書では規定しない
  - 満たすべき条件のみ記載: ①24時間稼働（送信スケジュール・フォローアップ・返信ポーリングが PC の電源に依存しない）②OAuth リダイレクトURI に到達できること ③DB とトークンのバックアップが取れること ④再起動後に自動復帰すること
- アプリDB: 当面は現行の **SQLite を継続**（単一サーバ配置なら十分・移行コストゼロ）。※「商用DB」（§4-F1）は Musubu 等の“企業リスト販売サービス”を指す用語であり、アプリのデータベース技術とは別物。複数拠点からの同時利用や Vercel 等のサーバレス配置に移る場合は Supabase（PostgreSQL）等への載せ替えを検討（Phase C 以降のオプション。スキーマは §5 のまま移行可能）
- Gmail API クォータは本規模では無視できる水準（200通/日 = 2万ユニット vs 上限8,000万）。※2026年後半にクォータ超過課金の導入予定が公式告知されているが本規模では影響なし

## 10. 開発順序 — 「最短で1通目を送る」を最優先に組む

### 設計の起点

**既存アプリは「URLを貼る → 分析 → メール生成 → 確認画面」まで完成している。** つまり**キャンペーン機能・リスト収集・一括生成を待つ必要はない**。確認画面に「送信」ボタンを付けるだけで、1通目を送り始められる。

この事実から、開発順序は **「送れる最小構成を最速で作り、送りながら残りを足す」** を原則とする。全機能を作り終えてから送り始める設計にはしない。

---

### Step 0: 下ごしらえ（**コード不要・並行実施**）

エンジニアの作業を待たずに進められるもの。**ここが未完だと Step 1 が完成しても送れない**ため、Step 1 と並行して片付ける。

| # | 作業 | 担当 | 備考 |
|---|------|------|------|
| 0-1 | **SPF / DKIM / DMARC を DNS に設定**（p=none から） | インフラ | **未設定だと何を作っても迷惑メール直行**。最重要 |
| 0-2 | Google Postmaster Tools にドメイン登録 | インフラ | 迷惑メール報告率 0.3% の監視用 |
| 0-3 | Google Cloud プロジェクト作成 → OAuth 同意画面を **Internal** に設定 | 蛯名 | Workspace 契約済みのため審査不要 |
| 0-4 | **送信担当者の確定**（U5）とアカウント準備 | ユーザー | 誰の Gmail から送るか |
| 0-5 | **署名の整備**（特電法の法定表示を含む） | ユーザー | 社名・住所・受信拒否の通知先・問合せ先。**これが無いと1通も合法に送れない** |
| 0-6 | 商材の実データ登録（サンプルを実物に差し替え） | ユーザー | 既存の /services 画面で可能 |
| 0-7 | テスト用の宛先アドレスを2〜3個用意 | ユーザー | Stage 1 の検証用 |
| 0-8 | **最初の宛先を20〜30社、手動で用意** | ユーザー | **商用DBの契約を待たない**。Google検索・手持ちの名刺で十分。まず送ることが目的 |

**💬 ざっくり**: プログラムを1行も書かずに進められる準備です。特に「DNSの設定」と「署名に会社の住所と配信停止先を入れること」は、これが無いと作っても送れない・送ると違法になる項目なので最優先です。

---

### Step 1: 「1通送れる」— 最小送信ライン 🎯 **ここがゴール**

既存の単発生成フローに送信機能を足すだけ。**キャンペーン・一括生成・リスト収集は一切作らない。**

| # | 実装 | 重さ | なぜ Step 1 に必要か |
|---|------|------|--------------------|
| 1-1 | `senders` テーブル + **Gmail OAuth 接続**（1名分でよい） | 中 | 送信の根幹 |
| 1-2 | **Gmail API 送信**（既存 `/prospect/[id]` の「Gmailで開く」を「送信」に置き換え） | 中 | `googleapis` + Nodemailer MailComposer |
| 1-3 | `send_log`（不変の送信ログ） | 軽 | 二重送信防止と監査の土台。**後から足すのは困難なので最初に入れる** |
| 1-4 | `suppressions` + **送信直前チェック** | 軽 | 配信停止依頼への対応は**法的義務**。無効化不可の固定仕様 |
| 1-5 | **「営業お断り」検出の警告表示** | 軽 | 既存 crawl の結果を使うだけ。特電法の例外要件（送ると違法になる） |
| 1-6 | **テストモード**（宛先を自分たちに強制上書き + 赤帯表示） | 軽 | **事故防止の最重要装置。実装が軽い割に守るものが大きい** |
| 1-7 | 送信成功で `send_status = sent` に自動更新 | 軽 | 手動プルダウンの廃止 |
| 1-8 | **出力スキーマ検証ゲート**（未解決の `{{変数}}` が残っていたら送信ブロック） | 軽 | ⚠️ **自社KBの `prompt-rebuild-field-dropout`（再発2回）と同型の事故**。実装が数行なのに、変数が露出したメールは一発で信用を失う |
| 1-9 | **自社・グループドメイン宛てのブロック** | 軽 | 誤送信ガードの最小版。5行で書ける |
| 1-10 | **List-Unsubscribe ヘッダの付与** | 軽 | 受信者がワンクリックで配信停止できる。**苦情率0.3%（=アカウント凍結ライン）を直接下げる**最も安価な手段 |
| 1-11 | **SQLite の日次バックアップ**（cron でファイルコピー） | 軽 | 単一ファイルDB + 単一ローカルサーバーは無保護だと**1回の事故で全損**。send_log は法的な証跡でもある |

**→ この時点で Stage 0（下書き）→ Stage 1（自分宛てテスト）→ Stage 2（実宛先へ1通ずつ承認送信）が回り始める。**

**Step 1 で意図的に作らないもの**: キャンペーン / 一括生成 / レビューキュー / リスト収集 / フォローアップ / 効果測定 / LP連携。**すべて後から足せる**。

---

### Step 2: 「毎日送れる」— 事故防止と量産

Step 1 で実際に送りながら、並行して作る。

| # | 実装 | 重さ | 理由 |
|---|------|------|------|
| 2-1 | **二重送信防止**（send_log 照合 + ユニーク制約 + 送信前の状態永続化） | 中 | 量が増えた瞬間に事故る |
| 2-2 | **既存顧客・取引先の除外**（F21） | 軽 | **既存客への「はじめまして」は一発で信用を失う。軽いのに損失が大きい** |
| 2-3 | **危険ワード・事実誤認の検知**（F18・特にハルシネーション照合） | 中 | AIが相手について嘘を書く事故を止める |
| 2-4 | 日次上限・送信間隔ランダム化・営業時間制御（F6 / F20） | 中 | 凍結回避。50通/日から漸増 |
| 2-5 | **CSV / Excel 取込**（F8） | 中 | 手動20社の限界を超える |
| 2-6 | **一括生成**（F5）+ **レビューキュー**（F7） | 重 | ここで初めて「1日50通」が現実になる |
| 2-7 | バウンス検知 + 健全性の自動停止（F6） | 中 | 気づかないうちの凍結を防ぐ |

**→ この時点で 1日50通の半自動運用が成立する。**

### Step 3: 「取りこぼさない」

| 実装 | 理由 |
|------|------|
| 返信検知 + `send_status` 自動化（F6） | 返信が来始めてから作れば間に合う |
| **Calendly Webhook**（F14） | 予約完了の自動検知 → ファネルが繋がる |
| 担当バッティング防止（F16） | **送信者が2人目になる前に入れる** |
| LINE 即時通知（F23 の即時分のみ） | 反応に即応できるようにする |
| 受信トレイ到達テスト（F29） | 「そもそも届いているか」を検証する |
| メール認証の自己診断（F30） | Step 0 の設定が正しいか継続確認 |

### Step 4: 「成果を上げる」

| 実装 | 理由 |
|------|------|
| **フォローアップシーケンス**（F12） | **返信率を最も押し上げる**。ただし1通目が安定してから |

### この洗い出しで見つかった重要な指摘

1. **⚠️ 自社KBの既知バグパターンが2件ヒット**
   - `prompt-rebuild-field-dropout`（再発2回）→ 差し込み変数の脱落。**Step 1-8 で機械検証する**
   - `lock-null-observation-false-negative` → レビュー画面の同時編集で上書き事故。**複数人でレビューを始める Step 2 で編集ロックを入れる**
2. **⚠️ 到達テストは Gmail だけでは足りない** — **日本のBtoBは Microsoft 365 の利用が多い**。F29 のシードアドレスに **Outlook / Microsoft 365 を必ず含める**（軽微な拡張だが、これが無いと相手先の迷惑メール判定を見落とす）
3. **⚠️ 「不変の send_log」と個人情報の削除請求が衝突する** — 個人情報保護法上の開示・削除請求に応じる義務があるため、send_log を完全不変にすると対応できない。**削除ではなく仮名化（宛先を復元不能な識別子に置換し、送信事実とログの整合は保つ）で対応する**設計とする（§11-L5 で弁護士確認）
4. **既存プロジェクトとの接続** — 「新規開業検知（Googleマップ等）」は**軽井沢の店舗LP営業**と直結する。「英語文面の自動切替」は**日英LP生成**を別途進めている以上、将来必ずぶつかる（現時点では対象外だが盲点として記録）
5. **矛盾の検出** — 候補にあった「動的パーソナライズ画像（相手のロゴを差し込んだ画像）」は **F9 の「相手の実ロゴ・実写真を使わない」方針と正面から矛盾する**ため、明確に不採用

### ⚠️ 要判断: 送信ドメインを分離するか

分析は「**送信専用サブドメイン**（例 `mail.cypherone.co.jp`）に分離し、評判失墜を本体ドメイン（採用・請求メール）から遮断すべき」と S 判定している。しかし:

- 本要件の中核価値は「**その人からのメール**として届くこと」。`shigenami@mail.cypherone.co.jp` は `@cypherone.co.jp` より**受信者から見て正当性が下がり、返信率を落としうる**
- ドメイン分離が定石とされるのは**大量一斉配信**の文脈。本ツールは**1日50〜200通・1対1・人間レビュー付き**であり、リスクプロファイルが異なる

**推奨（§9-4 の方針を維持）**: **当面は本体ドメイン `cypherone.co.jp` のまま送信する。** ウォームアップ・レビューキュー・苦情率監視で評判を守り、**Postmaster Tools の指標が悪化した時点でサブドメインへ退避する**（そのための移行手順だけ用意しておく）。→ 最終判断はユーザー（§11-U10）

---

### この順序で守っている原則

1. **Step 1 のスコープに「あれば便利」を1つも入れない** — 送れるまでの距離を最短にする
2. **後から足せないものだけ最初に入れる** — `send_log`（監査ログは遡って作れない）、suppressions（法的義務）、テストモード（事故ってからでは遅い）
3. **法令要件は Step 0/1 に前倒し** — 署名の法定表示・営業お断り検出・配信停止対応は、1通目の前に必ず揃える
4. **リスト調達を待たない** — 商用DBの契約・トライアル（U3）は Step 2-5 のタイミングでよい。最初の20〜30社は手動で用意する
5. **事故防止機能は「量が増える直前」に入れる** — 二重送信防止・既存顧客除外・バッティング防止は、それぞれ「量が増える」「顧客リストが混ざる」「送信者が2人になる」の直前

**💬 ざっくり**: 「全部作ってから送り始める」のをやめて、「送れる最小限だけ作って、送りながら残りを作る」順番にしました。既存のアプリはメールを作るところまで完成しているので、送信ボタンとログと安全装置だけ足せば1通目が出せます。リストも最初は手で30社集めれば十分で、有料データベースの契約は後回しで構いません。
